# Claude Project Guardrails — Charmster (SwiftUI)

## Non-negotiable safety rules

- Never delete, rename, or move files unless the user explicitly asks.
- Never perform sweeping refactors, reformatting, or "cleanup" passes unless asked.
- If a change will touch more than 10 files, stop and ask for confirmation before editing.
- Never rewrite published history on main:
  - No `git push --force`
  - No `git reset --hard` on commits that have been pushed
  - Prefer `git revert` for undoing pushed mistakes
- Before any push, show a summary:
  - `git diff --stat`
  - A 3–8 bullet summary of what changed and why

## Default Git workflow (always)

1. Sync main:
   ```
   git checkout main
   git pull --rebase
   ```
2. Create a branch for the change:
   ```
   git checkout -b fix/<short-description>
   ```
3. Make the smallest possible change to accomplish the task.
4. Verify with build/tests (see below).
5. Commit with a specific message.
6. Push branch (and open a PR if that's the team workflow).

## Build verification (SwiftUI)

Prefer CLI builds so failures are deterministic.

Always list schemes once if needed:
```
xcodebuild -list
```

Build for simulator (adjust scheme/device if needed):
```
xcodebuild -scheme "<SchemeName>" -destination 'platform=iOS Simulator,name=iPhone 17 Pro' clean build
```

If build fails:
- Stop at the first real error (not cascading errors)
- Fix minimally
- Re-run the same build command until it passes

> Note: `project.pbxproj` is gitignored and generated by XcodeGen. Run `xcodegen generate` inside `ios/` before building whenever source files are added or removed.

## Change discipline

- One task at a time.
- Keep commits small and auditable.
- Avoid touching `*.xcodeproj/project.pbxproj` unless necessary.
- If it must be changed, explain exactly why and keep the diff minimal.

## When to ask questions

Only ask a question if blocked. Otherwise proceed.

Examples of acceptable blockers:
- Missing scheme name / multiple schemes and unclear target
- Simulator device name differs from available devices
- A runtime config/env value is required (build ok but app crashes)

## Output format for each task

When finished, report:
- Commands run
- Build result (pass/fail)
- Files changed
- Next steps (only if asked)
